What changed from LLD v2.1 → v3.0

#ChangeWhy
1Claude Vision is no longer the default — the self-hosted, fine-tuned Qwen2.5-VL-7B (port :5002) is now the default provider. Both implementations coexist behind wisteria.vision.provider (vlm default · claude opt-in).Eliminates ~$0.003–0.006/query API cost by default. Consistent taxonomy vocabulary. No rate limits or external key required at runtime. Hosted Claude stays available for A/B comparison or fallback.
2Claude Haiku explanation call removed. Replaced by a deterministic template engine in ExplanationService.java.E-commerce users need consistent, trustworthy reasons — not variable prose. Template is a pure function of matched attributes: <1ms, $0, never fails.
3New vlm-server/ Python service (FastAPI) on port :5002. Added to Docker Compose alongside the existing CLIP server.Mirrors the CLIP server pattern already established in v2.1. Consistent deployment model.
4product_style_attributes schema gains confidence JSONB and model_ver VARCHAR(64) columns.Per-field confidence enables low-confidence logging and retraining triggers. model_ver enables the VLM version guard (mirrors the CLIP guard).
5Startup model-version guard extended to cover both CLIP (product_embeddings.model_ver) and VLM (product_style_attributes.model_ver).A VLM version mismatch between indexing and query produces attribute drift. Same failure mode as a CLIP mismatch.
6New optional POST /api/v1/style/chat endpoint backed by ChatService.java. LLM (Claude or self-hosted) is wired here only.Separates deterministic product matching from open-ended conversational reasoning. LLM is behind a feature flag; disabled by default.
7Degradation rule updated: VLM failure (not Claude failure) → embedding-only search + CLIP-only template. CLIP failure still → 503.VLM is now the secondary model. CLIP embedding is still the backbone; same rule, different model name.
8Latency budget updated: VLM extraction 0.2–1.5s (GPU) replaces Claude Vision 800–1200ms. Template explanation <1ms replaces Haiku 500–1000ms.Honest hardware-based estimates. Total cold query is GPU-bound, not API-round-trip-bound.
9training/ directory added. As-built, the distillation → fine-tune → eval → serve workflow is one command (run_pipeline.py) plus label_with_claude.py, fetch_catalog.py, and taxonomy.py (see build notes).Formalises the distillation → fine-tune → eval workflow as first-class project artifacts, not ad-hoc scripts.
10Evaluation plan updated: per-attribute VLM accuracy (≥85% gate) added as a pre-deployment check alongside the existing precision@8 ≥60% search quality gate.Two separate quality signals: model accuracy (classification) and end-to-end match quality (retrieval). Both must pass.
11New furniture gate (open-set rejection): CLIP server gains POST /classify (zero-shot image→text classification); the query path short-circuits non-furniture uploads to an out_of_domain response before the VLM runs. Knobs: FURNITURE_GATE_ENABLED, FURNITURE_GATE_THRESHOLD.The closed VLM vocabulary has no "not furniture" option, so out-of-domain images (e.g. an auto-rickshaw) hallucinated furniture attributes and scored 75. Image→image cosine can't separate them; image→text classification can (rickshaw 0.0004 vs chair 0.9998).
12Re-ranker adds a visual floor: top result's raw cosine < 0.50 also sets low_confidence. Explanations became confidence-aware — the template hedges its wording when the match is weak.Backstop for weak-but-in-domain matches that pass the gate, and honest copy that no longer overclaims "a close visual match" on low-confidence results.
13Response gains out_of_domain; the React ResultsGrid renders a distinct "not a furniture image" state (no banner, no score cards). Degraded (VLM-down) responses are no longer written to Redis.Non-furniture uploads now read as rejected rather than as low-confidence matches, and the cache no longer persists degraded results after the VLM recovers.
14Response gains vlm_degraded (set true only when the VLM was unavailable and the query ran embedding-only). ResultsGrid renders a separate degraded notice (.low-confidence.degraded) that takes precedence over the generic low-confidence copy.A transient system fault and a genuine weak-match verdict previously looked identical; users (and tests) can now tell them apart in both the API response and the UI. Makes the state-3 acceptance check (vlm_degraded: true) satisfiable.

Scenario

Customer uploads a room photo, Instagram screenshot, or magazine tearout showing a style they love — no keyword needed. The self-hosted fine-tuned VLM extracts style attributes (finish, material, silhouette, era, palette, mood). The image embedding is compared against Wisteria catalog vectors to find the closest aesthetic matches. Top-N matched products are shown with a similarity score and a deterministic "why this matches" note built from the overlapping attributes.

Python CLIP server (clip-server/main.py) — v3.0 adds the furniture gate

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import open_clip, torch, base64, io
from PIL import Image, ImageOps

app = FastAPI()
MODEL_VER = "ViT-L-14/openai"
model, _, preprocess = open_clip.create_model_and_transforms("ViT-L-14", pretrained="openai")
device = "mps" if torch.backends.mps.is_available() else "cpu"
model = model.to(device).eval()

class EmbedRequest(BaseModel):
    image_b64: str

@app.post("/embed")
def embed(req: EmbedRequest):
    try:
        img = Image.open(io.BytesIO(base64.b64decode(req.image_b64)))
        img = ImageOps.exif_transpose(img).convert("RGB")
    except Exception:
        raise HTTPException(400, "invalid_image")
    tensor = preprocess(img).unsqueeze(0).to(device)
    with torch.no_grad():
        vec = model.encode_image(tensor)
        vec = vec / vec.norm(dim=-1, keepdim=True)  # L2-normalise
    return {"embedding": vec.squeeze().cpu().tolist(), "model_ver": MODEL_VER}

# ── NEW in v3.0: zero-shot furniture gate (open-set rejection) ──
# Image→text CLIP classification. Far more discriminative for "is this
# furniture at all?" than image→image cosine, which cannot separate an
# auto-rickshaw (≈0.63 cosine to a teak chair) from a real chair.
tokenizer = open_clip.get_tokenizer("ViT-L-14")
FURNITURE_PROMPTS     = ["a photo of furniture", "a photo of a chair", "a photo of a sofa", ...]   # 9 prompts
NON_FURNITURE_PROMPTS = ["a photo of an auto rickshaw", "a photo of a vehicle", "a photo of a person", ...]  # 9 prompts
_ALL_PROMPTS = FURNITURE_PROMPTS + NON_FURNITURE_PROMPTS
_N_FURNITURE = len(FURNITURE_PROMPTS)
with torch.no_grad():                                  # precompute text features once at startup
    _tok = tokenizer(_ALL_PROMPTS).to(device)
    _text_features = model.encode_text(_tok)
    _text_features = _text_features / _text_features.norm(dim=-1, keepdim=True)
_logit_scale = model.logit_scale.exp()

@app.post("/classify")
def classify(req: EmbedRequest):
    img = _decode_image(req.image_b64)                  # shared EXIF-aware decoder (also used by /embed)
    tensor = preprocess(img).unsqueeze(0).to(device)
    with torch.no_grad():
        vec = model.encode_image(tensor)
        vec = vec / vec.norm(dim=-1, keepdim=True)
        logits = (_logit_scale * vec @ _text_features.T).squeeze(0)
        probs  = logits.softmax(dim=-1)                 # softmax over all 18 prompts
    furniture_prob = float(probs[:_N_FURNITURE].sum().item())   # mass on the furniture group
    top_idx = int(probs.argmax().item())
    return {"furniture_prob": furniture_prob, "top_label": _ALL_PROMPTS[top_idx],
            "top_is_furniture": top_idx < _N_FURNITURE, "model_ver": MODEL_VER}

@app.get("/health")
def health():
    return {"status": "ok", "device": device, "model_ver": MODEL_VER}

Python VLM server (vlm-server/main.py) — new in v3.0

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from transformers import AutoModelForImageTextToText, AutoProcessor
import torch, base64, io, json
from PIL import Image

app = FastAPI()
MODEL_DIR  = "./adapter/qwen2-vl-2b-wisteria-v1"    # LoRA-merged fine-tuned checkpoint (2B in the POC)
MODEL_VER  = "qwen2-vl-2b-wisteria-v1"
device     = "cuda" if torch.cuda.is_available() else "mps" if torch.backends.mps.is_available() else "cpu"

model     = AutoModelForImageTextToText.from_pretrained(
    MODEL_DIR, torch_dtype=torch.bfloat16, device_map=device)
processor = AutoProcessor.from_pretrained(MODEL_DIR)

# Closed taxonomy — model is trained on exactly these values
TAXONOMY = {
    "finish":    ["matte", "gloss", "brushed", "aged", "lacquered", "natural"],
    "material":  ["oak", "walnut", "velvet", "brass", "linen",
                  "marble", "rattan", "glass", "leather", "ceramic"],
    "silhouette":["clean-line", "curved", "ornate", "sculptural", "minimal"],
    "era":       ["mid-century", "art-deco", "japandi", "contemporary",
                  "industrial", "traditional"],
    "palette":   ["warm-neutral", "cool-neutral", "earthy", "monochrome", "bold"],
    "mood":      ["cosy", "editorial", "calm", "dramatic", "playful"],
}
SCHEMA_KEYS = ", ".join(TAXONOMY.keys())
PROMPT = (
    f"Extract interior-design style attributes as strict JSON with exactly these keys: "
    f"{SCHEMA_KEYS}. Use only the allowed taxonomy values for each key. "
    f"Output JSON only, no preamble."
)

class ExtractRequest(BaseModel):
    image_b64: str

def _parse_attrs(text: str) -> dict:
    start = text.index("{")
    end   = text.rindex("}") + 1
    attrs = json.loads(text[start:end])
    # Enforce closed vocabulary — unknown values fall back to closest known
    for key, allowed in TAXONOMY.items():
        if attrs.get(key) not in allowed:
            attrs[key] = allowed[0]   # safe default; log this in production
    return attrs

@app.post("/extract")
def extract(req: ExtractRequest):
    try:
        img = Image.open(io.BytesIO(base64.b64decode(req.image_b64))).convert("RGB")
    except Exception:
        raise HTTPException(400, "invalid_image")
    msgs = [{"role": "user", "content": [
        {"type": "image", "image": img},
        {"type": "text",  "text":  PROMPT}]}]
    inputs = processor.apply_chat_template(
        msgs, add_generation_prompt=True, tokenize=True,
        return_tensors="pt").to(device)
    with torch.no_grad():
        out = model.generate(**inputs, max_new_tokens=128, do_sample=False,
                             temperature=None, top_p=None)
    text  = processor.decode(out[0], skip_special_tokens=True)
    attrs = _parse_attrs(text)
    return {"attributes": attrs, "model_ver": MODEL_VER, "raw": text}

@app.get("/health")
def health():
    return {"status": "ok", "device": device, "model_ver": MODEL_VER}

Indexing flow

1
Load catalog manifest
CatalogIndexService reads the catalog manifest (CATALOG_PRODUCTS_FILE — file:/catalog/products.json from the read-only mount in Docker; classpath by default). Each entry: name, category, image_path (the product id is derived as a stable UUID from the image path). Triggered by POST /api/v1/admin/index, which starts a background virtual thread; an AtomicBoolean guard returns 409 if a run is already active.
products.json (mount)POST /admin/index → 202AtomicBoolean guard → 409
2
Read and normalize image bytes
Read raw bytes from disk (Files.readAllBytes()). Apply magic-byte validation (reject non-JPEG/PNG). Apply EXIF orientation correction. Scale shorter edge to 336, centre-crop to 336×336 using java.awt.Graphics2D. Base64-encode to a String b64. This encoded string is passed to both parallel calls. CLIP mean/std normalization happens inside the Python CLIP server only — Java must NOT attempt it.
magic-byte checkEXIF transpose336×336 centre-cropBase64
3
Parallel: CLIP embedding + style attributes
Two CompletableFutures on a virtual thread executor. (A) POST b64 to clip-server:5001/embed → returns {"embedding": float[768], "model_ver": "ViT-L-14/openai"}, 10s timeout. (B) Style attributes come from the pluggable CatalogAttributeSource: the default extractor source POSTs b64 to the VLM (:5002/extract → {"attributes": {...7 fields...}, "model_ver": "qwen2-vl-2b-wisteria-v1"}) with one retry then skip; the labels source (Docker default) instead returns the pre-computed Claude teacher label for the image — no network call. Joined with CompletableFuture.allOf(...).orTimeout(30, SECONDS) — bounded join prevents hanging the indexer. A single bad product is skipped-and-counted, never aborting the whole batch.
CLIP :5001attribute-source: labels | extractorallOf().orTimeout(30s)skip → count
4
Upsert into pgvector
Single @Transactional writes both tables. product_embeddings upsert writes the embedding vector and model_ver. product_style_attributes upsert writes all 6 attribute fields, the confidence JSONB (from the VLM response), model_ver, and the raw_json. HNSW handles incremental inserts natively — do NOT run REINDEX after every product. Only reindex after a bulk re-embedding with a new CLIP model version.
pgvector upsert@TransactionalON CONFLICT DO UPDATE
5
Log summary and expose status
Log total indexed, skipped, and elapsed time via @Slf4j. Write final status to IndexJobStatus singleton bean, exposed via GET /api/v1/admin/index-status.

ImageNormalizer responsibilities

StepDetail
Magic-byte validationVerify JPEG (FF D8 FF) or PNG (89 50 4E 47) signature — never trust Content-Type or extension. Reject others with 415. (Production images are JPEG/PNG only — WebP is not accepted.)
Size capReject >10 MB with 413 before reading into memory.
EXIF orientationApply exif_transpose equivalent in Java — phone photos arrive rotated. Missing this is a silent match-quality killer (silhouette is now upside-down).
Resize + centre-cropScale shorter edge to 336, centre-crop to 336×336 using java.awt.Graphics2D. Plain resize (v2.0 bug) stretches aspect ratio, distorting silhouette signals.
Base64 encodeJPEG quality ~85, Base64. Target <50 KB payload to both :5001 (CLIP) and :5002 (VLM). Same encoded string goes to both.

public Optional<StyleQueryResponse> probe(byte[] normalizedImageBytes) {
    long hash = PHashUtil.hash64(normalizedImageBytes);   // DCT-based 64-bit pHash
    String key = "style:query:" + Long.toHexString(hash);
    try {
        String cached = redisTemplate.opsForValue().get(key);
        if (cached != null) return Optional.of(objectMapper.readValue(cached, StyleQueryResponse.class));
    } catch (Exception e) {
        log.warn("Redis probe failed, continuing without cache", e);  // never fail the request
    }
    return Optional.empty();
}

public void write(byte[] imageBytes, StyleQueryResponse response) {
    String key = "style:query:" + Long.toHexString(PHashUtil.hash64(imageBytes));
    try {
        redisTemplate.opsForValue().set(key,
            objectMapper.writeValueAsString(response),
            Duration.ofSeconds(3600));
    } catch (Exception e) {
        log.warn("Redis write failed, continuing", e);
    }
}
✓
pHash tolerates re-uploads at different JPEG quality or size — a perceptually identical image produces the same hash even if the file bytes differ. TTL 3600 s, allkeys-lru, 256 MB cap (~25K entries). All Redis calls are wrapped in try-catch — Redis being down degrades to a full pipeline run, never a failed request.

Furniture gate — open-set rejection (new in v3.0)

The VLM works against a closed furniture vocabulary — it has no "not furniture" option, so when handed an out-of-domain image (an auto-rickshaw, a person, a landscape) it hallucinates plausible furniture attributes. Because attrOverlap carries 0.70 of the final score, those hallucinated attributes inflate the result: an auto-rickshaw was returning teak chairs at score 75. An image→image cosine floor cannot fix this — the rickshaw sits ≈0.63 cosine from a teak chair, inside the legitimate-furniture band (0.60–0.70). So we reject before the VLM/DB/re-rank work using a zero-shot image→text CLIP classifier.

ReRankingService — scoring model (as-built v3.0)

// Bounded convex combination — produces scores definitively in [0,1]
double cosineSim  = 1.0 - candidate.cosineDistance();   // [0,1], L2-normalised

// Weighted attribute overlap — fields are NOT equal weight (as-built)
double attrOverlap =
    0.40   * match("category",   query, candidate) +   // strongest signal
    0.25   * match("palette",    query, candidate) +
    0.20   * match("material",   query, candidate) +
    0.0375 * match("finish",     query, candidate) +   // each tail field weighted
    0.0375 * match("silhouette", query, candidate) +   // independently (4 × 0.0375 = 0.15)
    0.0375 * match("era",        query, candidate) +
    0.0375 * match("mood",       query, candidate);
// match() returns 1.0 on exact enum match, 0.0 otherwise; WEIGHTS sum to 1.0

double finalScore = 0.30 * cosineSim + 0.70 * attrOverlap;  // [0,1] — attributes dominate
int    apiScore   = Math.round((float)(finalScore * 100));   // 0–100

// Worked example (full pipeline, storage cabinet query):
// cosineSim=0.82, category+palette+material all match → attrOverlap≈0.85
// finalScore = 0.30×0.82 + 0.70×0.85 = 0.246 + 0.595 = 0.841 → score 84
// Scores in 76–91 range confirm attributes dominate when VLM runs cleanly.

// VLM-degraded path: extractQuietly() returned null → queryAttrs == null
// attrOverlap = 0.0 (no attributes) → finalScore = 0.30×cosineSim only
// For a typical cosineSim≈0.46: finalScore ≈ 0.138 → scores 13–23 → lowConfidence=true
// Both vlm_degraded=true AND low_confidence=true fire simultaneously.
// Degraded results are NOT written to Redis (avoids poisoning the cache).
if (vlmDegraded) {
    attrOverlap = 0.0;
    finalScore  = 0.30 * cosineSim;
}

// Visual floor (new in v3.0): even when attribute overlap inflates the score,
// reject as low-confidence if the TOP result's raw visual similarity is weak.
// Catches out-of-domain queries whose hallucinated attributes pushed the score up.
static final double VISUAL_FLOOR = 0.50;
double  topVisualSim = scored.isEmpty() ? 0.0 : 1.0 - scored.get(0).cosineDistance();
boolean lowConfidence = scored.isEmpty()
        || bestFinalScore < 0.45
        || topVisualSim < VISUAL_FLOOR;
✓
As-built weights: 0.30×cosine + 0.70×attrOverlap. Attributes dominate — category (0.40), palette (0.25), material (0.20), and finish/silhouette/era/mood (0.0375 each, summing to 0.15). This is the shipped configuration; the v2.1 design (0.65/0.35) was revised after observing that visual similarity alone produced too many off-category results. Low-confidence threshold (<0.45) sets low_confidence: true — fires on both genuine low-match queries and VLM-degraded queries (where attrOverlap=0 forces all scores into the 13–23 range). A second trigger, the visual floor (top result's raw cosine similarity < 0.50), also sets low_confidence even when attribute overlap inflated the score — a backstop for weak-but-in-domain matches that survive the upstream furniture gate.

ExplanationService — deterministic template engine (replaces Haiku)

// ExplanationService.java — pure Java, no API call, O(1) per product
public class ExplanationService {

    // Priority order for attribute mention — most visually salient first
    private static final List<String> PRIORITY =
        List.of("era", "material", "palette", "finish", "silhouette", "mood");

    public String render(StyleAttributes query, CandidateProduct product) {
        List<String> matched = PRIORITY.stream()
            .filter(f -> Objects.equals(query.get(f), product.get(f)))
            .limit(3)
            .toList();

        if (matched.isEmpty()) return "Strong visual match across overall style.";

        return switch (matched.size()) {
            case 1 -> String.format(
                "Shares your inspiration's %s aesthetic.",
                label(matched.get(0), product));
            case 2 -> String.format(
                "Echoes your inspiration through its %s and %s.",
                label(matched.get(0), product),
                label(matched.get(1), product));
            default -> String.format(
                "Captures your inspiration's %s feel with a %s %s.",
                label("era", product),
                label("material", product),
                label("finish", product));
        };
    }

    // Formats a field into readable prose: "material=oak" → "oak material"
    private String label(String field, CandidateProduct p) {
        return p.get(field) + " " + field;
    }

    public Map<UUID, String> renderBatch(
            StyleAttributes query,
            List<CandidateProduct> products) {
        return products.stream().collect(
            Collectors.toMap(CandidateProduct::productId, p -> render(query, p)));
    }
}
✓
The template engine has no failure path — it is a pure function of data. No timeout, no fallback, no API dependency. If the VLM was degraded, the template renders using whichever attributes are available (could be zero, in which case it returns the CLIP-only variant). Explanations are consistent across the result grid because they all share the same matched-field logic. Confidence-aware phrasing (v3.0): the explainer now takes a lowConfidence flag and hedges its wording accordingly — a confident match reads "echoes your inspiration's … carrying the same overall character," while a low-confidence match reads "one of the closest we found … though the overall look may differ." This avoids overclaiming on weak matches.

Full orchestration — StyleQueryService.java

public StyleQueryResponse query(MultipartFile image, int topK,
                                   String filterEra, String filterMaterial) {
    long start = System.currentTimeMillis();

    // 1. Normalize image
    byte[] normalized = imageNormalizer.normalize(image);
    String b64        = Base64.getEncoder().encodeToString(normalized);

    // 2. Cache probe
    Optional<StyleQueryResponse> cached = cacheService.probe(normalized);
    if (cached.isPresent()) return cached.get().withCacheHit(true);

    // 3. Parallel fan-out: CLIP embed + furniture gate + VLM
    var executor   = Executors.newVirtualThreadPerTaskExecutor();
    var embFuture  = CompletableFuture.supplyAsync(() -> clipService.embed(b64), executor);
    var gateFuture = CompletableFuture.supplyAsync(() -> clipService.classify(b64), executor);
    var attrFuture = CompletableFuture.supplyAsync(() -> vlmService.extract(b64), executor);

    // 3a. Furniture gate — short-circuit out-of-domain BEFORE VLM/DB/re-rank
    var gate = awaitGate(gateFuture);                  // degrades OPEN on /classify failure
    if (gateEnabled && gate.furnitureProb() < gateThreshold) {
        attrFuture.cancel(true);
        return outOfDomainResponse(start);             // out_of_domain=true, no results
    }

    CompletableFuture.allOf(embFuture, attrFuture).orTimeout(20, SECONDS).join();
    float[]       queryVector = embFuture.get();
    StyleAttributes queryAttrs  = attrFuture.isCompletedExceptionally() ? null : attrFuture.get();
    boolean       vlmDegraded  = (queryAttrs == null);

    // 4. ANN search + optional pre-filter
    var candidates = embeddingRepo.findTop50(toVec(queryVector));

    // 5. Re-rank → Top 8
    var top8 = reRankingService.rank(candidates, queryAttrs, topK, vlmDegraded);

    // 6. Template explanations — synchronous, O(N), never fails
    var explanations = explanationService.renderBatch(queryAttrs, top8);

    // 7. Build response, write cache, log query
    var response = buildResponse(top8, explanations, queryAttrs, vlmDegraded,
                                 System.currentTimeMillis() - start);
    cacheService.write(normalized, response);
    queryLogRepo.save(new StyleQueryLog(response, vlmDegraded));
    return response;
}

v3.0 build notes

Refinements made during implementation. The v2.1 (Claude) and v3.0 (self-hosted) paths coexist behind config switches; the v3.0 self-hosted stack is the default.
provider switches coexistence S3 sourcing

Pluggable providers

Vision — wisteria.vision.provider

vlm (default) → self-hosted Qwen2.5-VL on :5002 · claude → Claude Vision. One StyleExtractor seam, two implementations (service/vision/vlm, service/vision/claude), exactly one active via @ConditionalOnProperty.

Explanation — wisteria.explain.provider

template (default) → deterministic engine · claude → Haiku. One Explainer seam (service/explain/template, service/explain/claude). Hosted providers are retained behind the switches, not deleted.

Index attributes — wisteria.catalog.attribute-source

extractor (default) → live StyleExtractor (VLM/Claude) per image · labels → read the Claude teacher labels from labeled.jsonl. One CatalogAttributeSource seam (service/index), two implementations, @ConditionalOnProperty. Index-path only — the online query path always uses the configured vision provider.

Indexing without the VLM — the labels attribute source

The offline indexer no longer hard-depends on a running VLM. A new CatalogAttributeSource seam decouples index-time attribute extraction from the live query path. With wisteria.catalog.attribute-source=labels (the Docker Compose default), LabelsAttributeSource loads the Claude teacher labels (training/data/labeled.jsonl, copied into the read-only /catalog mount as labeled.jsonl) once at startup into a basename → StyleAttributes map, mapping the label category field onto furnitureCategory exactly as the Claude extractor does. Indexing then needs zero API calls and no VLM/Colab tunnel — CLIP embeddings still come from the clip-server, only the style attributes change source. The default extractor source preserves the original v3.0 behaviour (live VLM/Claude, one retry then skip-and-count). This eliminates the earlier failure mode where a dead :5002 tunnel caused the indexer to skip every product: the gold labels already paid for during distillation are reused as the catalog's ground-truth attributes (teacher = ground truth).

Training pipeline (as-built)

The three planned scripts (label_with_claude.py, train_lora.py, eval_golden.py) were consolidated into a single one-command pipeline plus two helpers: run_pipeline.py (label → train → eval(held-out) → serve, stages toggled by --skip-* / --serve; runs on a Colab/GPU T4), fetch_catalog.py (category-balanced catalog builder), and taxonomy.py (the single-source closed vocabulary, kept identical to the VLM server and the Java tool schema). As-built deltas from the design:

  • Student model: Qwen2-VL-2B-Instruct (LoRA, ~0.2% trainable params), downsized from the 7B design target to fit a free Colab T4. VLM_BASE_MODEL restores 7B where GPU memory allows.
  • Teacher labeling via the Anthropic Message Batches API (claude-sonnet-4-6, forced tool use, standard mode) — 50% off list price, checkpointed per batch, incremental by criteria_version so unchanged images are never re-billed.
  • Catalog: 462 category-balanced images scraped additively from wisteria.com (bed short at 36 — the site ran out), JPEG/PNG only. Labeling spend recorded: ~$0.37 (96 imgs) + ~$1.43 (366 imgs), batch-discounted.
  • Held-out eval gate relaxed to overall ≥40% at POC data scale (advisory — never blocks --serve); per-attribute ≥85% remains the production target. Latest run: 67.2% overall (category 95.7%, abstract fields palette/era/silhouette 52–62%).
  • Serve exposes /extract + /health on :5002 behind a cloudflared tunnel; paste the printed URL into the app's VLM_SERVER_URL.
