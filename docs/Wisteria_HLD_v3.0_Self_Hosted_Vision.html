High Level Design

Wisteria — Visual Style Matching

Upload a room photo, Instagram screenshot, or magazine scan. CLIP captures visual style; a fine-tuned, self-hosted vision model (Qwen2.5-VL) extracts the style taxonomy; vector similarity returns the closest Wisteria catalog products — each with a deterministic, template-built explanation of why it matches. No third-party LLM sits on the per-query critical path.
v3.0 — Self-hosted vision Spring Boot 3.x React 18 CLIP ViT-L/14 Fine-tuned Qwen2.5-VL pgvector Redis Docker Compose
✓
Goal: Deliver robust visual search where every per-query model runs on our own infrastructure. Upload an image, extract style attributes using a fine-tuned, self-hosted VLM, generate CLIP embeddings, perform ANN search using pgvector HNSW, apply in-memory attribute boosting, and build explanations deterministically from the matched attributes.
⚠
What changed from v2.1 → v3.0: Claude Vision is removed from the per-query path and replaced by a fine-tuned Qwen2.5-VL attribute extractor (Florence-2 as a lighter alternative). Claude Haiku explanations are replaced by a deterministic template engine. An LLM is now optional and reserved for a separate conversational endpoint. Claude's only remaining role is offline: as the teacher that bootstraps the VLM's training labels (Phase 1). CLIP, pgvector, Spring Boot, re-ranking and the API contract are unchanged.

CLIP generates a 768-dimensional vector while the fine-tuned Qwen2.5-VL extracts material, finish, silhouette, era, palette and mood as strict JSON. Both models are self-hosted.

6

Final ranked results and explanations are returned synchronously to the React UI.

What runs on your local machine

Five processes share localhost via Docker Compose. The React dev server is the only process outside Docker, running on port 3000 with Vite's proxy forwarding all /api calls to Spring Boot on 8080. Both model servers (CLIP and the fine-tuned VLM) are self-hosted — nothing on the per-query path leaves the machine.

Design decisions

Why CLIP ViT-L/14?

CLIP is trained on image-text pairs, so its embedding space naturally encodes aesthetic style rather than object identity. A ResNet backbone would cluster by object type (chair, lamp, table); CLIP clusters by visual style (Japandi, mid-century, ornate). Same model version must be used for both catalog indexing and query-time embedding — a version mismatch corrupts the embedding space.

Why pgvector over a dedicated vector DB?

For a POC catalog of 20–500 products, pgvector with HNSW is more than sufficient. It keeps the stack to a single Docker container and lets us use Spring Data JPA natively. The production migration path is clear: swap the repository layer for a Pinecone client when the catalog exceeds ~2M items or query volume demands sub-10ms ANN latency.

Why parallel fan-out with virtual threads?

CLIP embedding (~3–6s on CPU) and Claude Vision attribute extraction (~800ms) are independent. Running them sequentially would make the pipeline 4–7s. Java 21 virtual threads make CompletableFuture.allOf() extremely cheap — no thread pool sizing needed. Total latency is bounded by the slower call (CLIP on CPU), not their sum.

Why pHash for Redis caching?

Perceptual hashing (DCT-based, 64-bit) allows cache hits for the same inspiration image even if re-uploaded at a slightly different size or JPEG quality. A byte-exact hash would miss these. A Redis cache hit bypasses the entire pipeline and responds in under 80ms — important for demo scenarios where the same image is uploaded multiple times.

Why a fine-tuned VLM over Claude Vision?

Attribute extraction here is a fixed, closed taxonomy (material / finish / silhouette / era / palette / mood) — a classification problem, not open reasoning. A fine-tuned Qwen2.5-VL matches Claude's output on our own labels while running self-hosted: no per-query API cost, no external rate limit, predictable latency, and consistent terminology. Florence-2 is the lighter alternative when GPU budget is tight. Claude remains the teacher that produces the training labels offline.

Why deterministic template explanations?

E-commerce users need consistent, trustworthy reasons a product matched — not creative prose. Once the matched attributes are known (oak · Japandi · warm-neutral), the explanation is a pure function of that data, so a template engine renders it in <1 ms with zero variance and zero cost. Genuinely creative or conversational reasoning ("which of these brightens a dark room?") is routed to the optional /style/chat LLM endpoint instead.

Cost levers

Cache hit rate (biggest)

Every Redis pHash hit saves one CLIP + VLM call. At 30% → 60%, you halve GPU power spend. Tune TTL (3600 s) and 256 MB cap. A "warm" cache from prior queries is free.

Model size choice

Florence-2 is 0.77B or 2.3B, runs on 8–16GB GPUs; Qwen2.5-VL-7B needs 24GB. Florence-2 is 3–5× cheaper on GPU but slightly lower accuracy. Measure on your golden set before deciding.

GPU location

Local RTX 4090: ~$2–8k purchase, then marginal cost is electricity (~$10–20/mo). Cloud GPU: $250–400/mo. Local wins if you operate >6 months. Spot instances cut cloud cost by 70%.

Training data quality

Better labels = fewer retraining cycles. A 5k golden set from Claude labels well; bad labels force monthly retraining. Invest in labeling upfront.

Offline pipeline

Catalog indexing

Reads a local product catalog JSON, generates CLIP embeddings and style attributes (via the self-hosted fine-tuned VLM) for each product, and upserts both into pgvector. Triggered once at setup, or on-demand via REST. Note: in Phase 1, before the VLM is trained, this same job calls Claude to generate the silver labels that become the VLM's training set — see the Training & distillation section.
⚠
The CLIP model takes 10–15 s to load on first call. Subsequent calls run in 3–6 s on CPU (Intel/AMD) or 0.5–1 s on Apple Silicon MPS. Indexing 50 products takes roughly 5–8 minutes on CPU. Run indexing once before the demo.

Pipeline steps

1
Load catalog from local JSON
CatalogIndexService reads src/main/resources/catalog/products.json from classpath. Each entry contains id, name, category, and imagePath (relative to /tmp/wisteria-catalog/). The CommandLineRunner bean activates only when the index Spring profile is active.
products.jsonCommandLineRunner--spring.profiles.active=index
2
Read and normalize image bytes
Read raw image bytes from disk with Files.readAllBytes(). Apply EXIF orientation correction, scale the shorter edge to 336 and centre-crop to 336×336 using java.awt.Graphics2D (no external library; plain resize would stretch aspect ratio and distort silhouette signals). Encode to Base64 for HTTP transport. CLIP mean/std normalization happens exclusively inside the Python server's preprocess transform — Java must not attempt it.
java.awt336×336Base64
3
Parallel: CLIP embedding + VLM attribute extraction
Two CompletableFutures run in parallel using a virtual thread executor. (A) POST Base64 image to localhost:5001/embed — returns float[768]. (B) POST image to the VLM server at localhost:5002/extract — returns JSON with finish, material, silhouette, era, palette, mood plus per-field confidence. Both joined with CompletableFuture.allOf(). One retry on malformed VLM response; skip product on second failure.
CLIP :5001Qwen2.5-VL :5002CompletableFuture.allOf()virtual threads
4
Upsert into pgvector
Spring Data JPA native query: INSERT INTO product_embeddings ... ON CONFLICT (product_id) DO UPDATE SET embedding = EXCLUDED.embedding. Separate upsert into product_style_attributes. Both wrapped in a single @Transactional. Also persist model_ver per row. HNSW handles incremental inserts natively — reindex (REINDEX INDEX CONCURRENTLY product_embeddings_hnsw_idx) only after a bulk re-embedding with a new CLIP model version, not after every run.
pgvectorSpring Data JPAON CONFLICT DO UPDATEHNSW reindex
5
Log summary and expose status
Log total indexed, skipped, and elapsed time via @Slf4j. Write final status to IndexJobStatus singleton bean, exposed via GET /api/v1/admin/index-status so the React admin panel (or curl) can poll progress.

Python CLIP server

# clip-server/main.py
from fastapi import FastAPI
from pydantic import BaseModel
import open_clip, torch, base64
from PIL import Image
import io

app = FastAPI()
model, _, preprocess = open_clip.create_model_and_transforms(
    'ViT-L-14', pretrained='openai'
)
model.eval()
# Use MPS on Apple Silicon for 5-10x speedup over CPU
device = "mps" if torch.backends.mps.is_available() else "cpu"
model = model.to(device)

class EmbedRequest(BaseModel):
    image_b64: str

@app.post("/embed")
def embed(req: EmbedRequest):
    img_bytes = base64.b64decode(req.image_b64)
    img = Image.open(io.BytesIO(img_bytes)).convert("RGB")
    tensor = preprocess(img).unsqueeze(0).to(device)
    with torch.no_grad():
        vec = model.encode_image(tensor)
        vec = vec / vec.norm(dim=-1, keepdim=True)  # L2 normalise
    return {"embedding": vec.squeeze().cpu().tolist()}

@app.get("/health")
def health():
    return {"status": "ok", "device": device}

Python VLM server (fine-tuned Qwen2.5-VL)

# vlm-server/main.py — serves the LoRA-merged attribute extractor
from fastapi import FastAPI
from pydantic import BaseModel
from transformers import AutoModelForImageTextToText, AutoProcessor
import torch, base64, io, json
from PIL import Image

app = FastAPI()
MODEL_DIR = "./adapter/qwen2.5-vl-7b-wisteria"   # LoRA merged into base
MODEL_VER = "qwen2.5-vl-7b-wisteria-v1"
device = "cuda" if torch.cuda.is_available() else "cpu"

model = AutoModelForImageTextToText.from_pretrained(
    MODEL_DIR, torch_dtype=torch.bfloat16, device_map=device)
processor = AutoProcessor.from_pretrained(MODEL_DIR)

# Closed taxonomy — the model is constrained to these values
SCHEMA = "finish, material, silhouette, era, palette, mood"
PROMPT = ("Extract the interior-design style attributes as strict JSON "
          f"with exactly these keys: {SCHEMA}. Use only catalog vocabulary.")

class ExtractRequest(BaseModel):
    image_b64: str

@app.post("/extract")
def extract(req: ExtractRequest):
    img = Image.open(io.BytesIO(base64.b64decode(req.image_b64))).convert("RGB")
    msgs = [{"role": "user", "content": [
        {"type": "image", "image": img},
        {"type": "text", "text": PROMPT}]}]
    inputs = processor.apply_chat_template(
        msgs, add_generation_prompt=True, tokenize=True,
        return_tensors="pt").to(device)
    with torch.no_grad():
        out = model.generate(**inputs, max_new_tokens=128, do_sample=False)
    text = processor.decode(out[0], skip_special_tokens=True)
    attrs = json.loads(text[text.index("{"):text.rindex("}")+1])
    return {"attributes": attrs, "model_ver": MODEL_VER}

@app.get("/health")
def health():
    return {"status": "ok", "device": device, "model_ver": MODEL_VER}

Compute 64-bit pHash. Probe Redis: redisTemplate.opsForValue().get("style:query:" + phashHex). Cache hit → return immediately.
React FormDataMultipartFileRedis pHash probe
2
Image pre-processing
Validate by magic bytes — never trust Content-Type or extension (JPEG/PNG/WebP only, else 415; >10 MB rejected with 413 before buffering). Apply EXIF orientation correction (phone photos arrive rotated otherwise — a silent match-quality killer). Scale shorter edge to 336, centre-crop to 336×336 using java.awt.Graphics2D. Base64-encode. This encoded string is passed to both parallel calls in step 3.

3
Parallel fan-out — CLIP embedding + attribute extraction
Two CompletableFutures launched on a virtual thread executor. (A) ClipEmbeddingService.embed(base64) — POST to localhost:5001/embed, returns float[768], 10 s timeout. (B) AttributeExtractService.extract(base64) — WebClient POST to the self-hosted VLM at localhost:5002/extract, returns the attribute map + confidences. Joined with CompletableFuture.allOf(futA, futB).join().
CLIP :5001 ~3–6 sQwen2.5-VL :5002 ~0.2–1.5 svirtual thread executor
4
ANN cosine search with optional attribute pre-filter
First execute SET LOCAL hnsw.ef_search = 50 in the same transaction — pgvector's default ef_search = 40 silently degrades recall when LIMIT exceeds it. Then the Spring Data JPA native query: SELECT pe.product_id, (pe.embedding <=> CAST(:vec AS vector)) AS dist FROM product_embeddings pe ORDER BY dist ASC LIMIT 50. If era or material attribute is selective (expected match rate <20%), add a JOIN with product_style_attributes and a WHERE clause before the <=> operator to reduce search space. Returns 50 candidates with raw cosine distances.
pgvector <=> cosinetop-50 candidatesoptional attr pre-filter
5
Weighted re-ranking (in-memory Java)
Java stream over 50 candidates: score = 0.65 × (1 - cosineDist) + 0.35 × attributeOverlap. attributeOverlap counts matching fields between query attributes and product attributes, normalised to 0–1 over 6 fields. Sort descending, take top 8. Entirely in-memory — no additional DB round-trip. If the best final score is below 0.45, set low_confidence: true in the response so the UI frames results as "closest matches we found" rather than implying strong matches.
α=0.65 cosine + β=0.35 attributetop-8 selected
6
Template explanation assembly (deterministic, in-memory)
For each of the 8 results, ExplanationService renders a 1–2 sentence explanation from the overlap between query attributes and product attributes — no model call. A slotted template selects phrasing by which fields matched, e.g. "Shares your inspiration's {era} aesthetic through its {material} {finish} finish and {palette} palette." Empty slots are dropped gracefully so sparse matches still read naturally. Because it is a pure function of the matched data, output is consistent and effectively instant. Open-ended reasoning is out of scope here and is handled by the optional /style/chat endpoint.
template enginedeterministic<1 ms

Assemble response, cache, return
Build StyleQueryResponse POJO. Normalise cosine distance: score = Math.round((1 - cosineDist) * 100). Serialise to JSON. Write to Redis under style:query:{phashHex} with TTL 3600 s. Log query to style_query_log table. Return 200 JSON to React.

Model Layer

Training & distillation

The path from v2.1 (Claude Vision) to v3.0 (fine-tuned Qwen2.5-VL) is a three-phase ladder. Claude is the teacher; the VLM is the student. Investment is front-loaded (Phase 1–2); payoff is perpetual (Phase 3 onward).
Phase 1 — Distillation Phase 2 — Fine-tune Phase 3 — Production

The three-phase roadmap

1
Phase 1 — Distillation (one week)
Goal: Build a labeled training dataset by running your catalog + a sampling of real query images through Claude Sonnet 4.6 (or higher) as the teacher model. Sonnet's stronger steerability and tool-use adherence keep every label inside the closed attribute vocabulary — that fidelity is what makes a silver set clean enough to distill from. This produces a silver set of 5k–10k labeled images. Inputs: your products.json, 20–50 query images from users/demos. Process: training/label_with_claude.py posts each image to the teacher with forced tool use via the Anthropic Batch API (50% discount vs. real-time calls) and parses the extracted attributes into a JSONL training file. Output: wisteria_train_silver.jsonl (5k–10k rows). Cost: ~$1–2 per 500 images via Batch API; ~$9–10 for a full 5k-image set (vs. ~$15–30 at real-time pricing). As-built: 462 images cost ~$1.80 total. Quality gate: have domain experts hand-review a 5% sample and flag any obvious errors (e.g. "leather" labeled as "fabric"). Acceptable error rate: <5% across your taxonomy.
Claude Sonnet 4.6 — Teacher ModelAnthropic Batch API (50% off)~$9–10 / 5k images
2
Phase 2 — LoRA fine-tune (one week)
Goal: Fine-tune a base Qwen2.5-VL-7B on your silver set using parameter-efficient PEFT/LoRA. This trains only the adapter weights (~5% of model), leaving the backbone frozen. Process: training/train_lora.py runs on your GPU (RTX 4090, 1×24GB) or cloud compute. 5k images, 3 epochs, mixed-precision training (bfloat16) takes ~2–4 hours. Output: merged LoRA checkpoint saved to vlm-server/adapter/qwen2.5-vl-7b-wisteria-v1/. Eval: Run training/eval_golden.py on a hand-reviewed golden set of 200–500 images (Yogesh's team labels these in parallel with Phase 1). Measure per-attribute accuracy. Gate: Material, finish, era, palette must each reach ≥85% accuracy. If not, iterate the training (higher LR, more epochs, or better data curation).
PEFT LoRA2–4 hrs GPU~$5–20 (cloud)per-attr eval
3
Phase 3 — Production cutover
At gate: Phase 2 eval ≥85% per-attribute accuracy, golden-set precision@8 (search + re-rank + template) ≥60%. Cutover steps: (1) Merge the LoRA adapter into the base Qwen2.5-VL checkpoint (one-liner via transformers CLI). (2) Copy the merged checkpoint to vlm-server/adapter/qwen2.5-vl-7b-wisteria-v1/. (3) Restart vlm-server container. (4) Re-index the catalog with the new VLM (CatalogIndexService detects the model_ver change and runs full extraction). (5) A/B test the old indexing against the new — verify latency, match quality, and downstream metrics (click-through, conversion). (6) Sunset the Claude Vision calls in production (keep the ANTHROPIC_API_KEY only for the optional /style/chat endpoint). Ongoing: log low-confidence extractions (the confidence JSONB from the VLM) and roll them into a monthly re-training pipeline.
LoRA mergemodel cutoverre-indexA/B test

Dataset specifications

Phase 1 silver set (5k–10k images)

Source: 100% of your catalog products + a sampling of real query inspiration images. Format: one JSONL per image: {"image_b64": "...", "finish": "matte", "material": "oak", ...}. Labels: generated by Claude Sonnet 4.6 (forced tool use, one call per image). Quality: silver = Claude's output as-is. No human review except a 5% sample check for obvious errors. Size: 5k images is solid for 80%+ accuracy; 10k is ideal for ≥85%.

Phase 2 golden set (200–500 images)

Source: A balanced stratified sample from Phase 1 (overrepresent rare eras / materials). Labels: 100% hand-reviewed and corrected by domain experts (Yogesh's team or interior designers). Format: same JSONL, but each label is authoritative. Splits: 70% train (reused from silver, but corrected), 30% eval (held-out golden set). Use: train/eval gold to compute per-attribute accuracy. If accuracy <85%, curate a smaller 1k-image "hard negative" set (edge cases) and re-train.

Training code sketch

# training/train_lora.py — simplified LoRA fine-tune
from peft import get_peft_model, LoraConfig, TaskType
from transformers import AutoModelForVision2Seq, AutoProcessor, Trainer, TrainingArguments
from datasets import load_dataset
import torch

# Load base model + processor
model = AutoModelForVision2Seq.from_pretrained(
    "Qwen/Qwen2.5-VL-7B-Instruct", torch_dtype=torch.bfloat16, device_map="auto")
processor = AutoProcessor.from_pretrained("Qwen/Qwen2.5-VL-7B-Instruct")

# LoRA config
peft_cfg = LoraConfig(task_type=TaskType.CAUSAL_LM, r=64, lora_alpha=128, lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"])
model = get_peft_model(model, peft_cfg)

# Load silver set
train_ds = load_dataset("json", data_files="wisteria_train_silver.jsonl")["train"]

# Training
trainer = Trainer(
    model=model, args=TrainingArguments(
        output_dir="./checkpoints", num_train_epochs=3, per_device_train_batch_size=4,
        learning_rate=2e-4, bf16=True),
    train_dataset=train_ds)
trainer.train()

# Merge & save
model = model.merge_and_unload()
model.save_pretrained("./vlm-server/adapter/qwen2.5-vl-7b-wisteria-v1")

When to retrain

Monthly re-training (recommended)Sample 500 low-confidence extractions from the style_query_log, have domain experts correct them, append to the training set, and run a 1-epoch fine-tune. Merge and roll out Friday evening.
Major catalog updateNew product category (e.g., outdoor / garden) enters the catalog. Curate 200–300 labeled images from that category, add to training set, re-train with 2–3 epochs.
Model drift detectedIf golden-set precision@8 drops >10% over a quarter, or per-attribute accuracy on a held-out monthly sample drops <80%, trigger an urgent re-training cycle.
New base model releaseQwen2.5-VL-8B or Qwen3-VL released with better performance. Download the new base, apply the same LoRA config, initialize from the old checkpoint (transfer learning), fine-tune 1–2 epochs on the golden set to warm-start.

VLM server down / extraction timeout Medium Fall back to embedding-only ANN search (skip attribute boosting). Return results with why_matches set to a CLIP-only template ("Strong visual match"). Self-hosted, so no key/quota failure class — only process/health failures, surfaced via /health and a circuit breaker on the VLM WebClient.
pgvector not available (Docker not started) High Spring Boot fails fast on startup with a clear error. Docker Compose healthcheck on postgres prevents the API from starting until the DB is ready when using depends_on: condition: service_healthy.
Redis not available Low Cache is non-blocking. CacheService wraps all Redis calls in try-catch. On any Redis error, skip cache probe and write, run full pipeline, log a warning. Functional degradation only (slower, no caching).
Sparse match — few/no overlapping attributes Low Template engine drops empty slots and falls back through a priority order (era → material → palette → "visually similar"). It can never produce malformed or off-tone text because output is a pure function of the matched fields. No retry path needed.
VLM taxonomy gap / model drift over time Medium New catalog styles outside the trained vocabulary degrade silently. Mitigation: log low-confidence extractions (from the confidence JSONB), sample them into the next labeling round, and schedule periodic re-training. Track per-attribute accuracy on the golden set as a release gate.
Model version mismatch (CLIP or VLM re-trained) High Store model_ver in both product_embeddings (CLIP) and product_style_attributes (VLM). On startup, CatalogIndexService checks all indexed rows use the current versions. A CLIP change forces re-embedding; a VLM change forces re-extraction. Mismatch logs a warning and blocks queries until re-indexed.
Poor match quality on abstract images (lifestyle shots) Low Accept for POC. Production mitigation: add a zero-shot image type classifier to detect lifestyle vs product photo and adjust the Claude Vision prompt framing. Set "inspiration_mode": true in response so the UI can set expectations.

Quality evaluation — proving "closest aesthetic matches"

The POC isn't done when the plumbing works; it's done when match quality is demonstrated. Before the demo: build a golden set of 15 inspiration images mirroring the scenario verbatim — 5 clean product shots, 5 room scenes, 3 Instagram screenshots (with UI chrome), 2 magazine scans. For each, a human marks every returned product relevant or irrelevant. Sign-off target: precision@8 ≥ 60%. Grid-search the re-ranking weights α ∈ {0.5, 0.65, 0.8} against this set and record the chosen values in the README. Pull p50/p95 latency from style_query_log across 30 queries. Cost check: $0 per-query API cost — self-hosted Qwen2.5-VL + CLIP fanned out on the virtual-thread executor; the only model spend is one-time Phase 1 distillation labeling, and indexing is local compute. (Running the optional Claude vision/explanation providers instead would add ≈ $0.01–0.03 per cold query.)

Production migration path

When to migrate off local

The POC is ready to move to production infrastructure when: (1) the core embedding + search + explanation quality is validated with real Wisteria catalog images, (2) the re-ranking weights (α, β) have been tuned via user feedback, and (3) the API contract is stable.

What changes in production

Swap ClipEmbeddingService WebClient URL from localhost:5001 to the SageMaker endpoint URL. Swap Files.copy() image save for an S3 presigned upload. Swap the Docker pgvector container for RDS. Swap local Redis for ElastiCache. Spring service layer code does not change.
